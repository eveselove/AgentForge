# План рефакторинга маршрутизации и исполнения агентов AgentForge

**Дата:** 2026-06-XX  
**Статус:** В процессе реализации (Фаза 1 начата 2026-06)  
**Цель:** Сделать так, чтобы мощные исполнители (Grok и Jules) реально использовались, а Antigravity получал задачи, которые он может осмысленно обрабатывать (или явно делегировал).

---

## 1. Диагноз текущего состояния (кратко)

### Проблемы

1. **Сильный перекос в routing**
   - `resolve_agent()` в `task_queue.py` и многие `create_*.py` скрипты по умолчанию шлют сложные/архитектурные/аналитические задачи на `antigravity`.
   - В результате Grok и Jules голодают.

2. **Antigravity почти не исполняется автоматически**
   - В `dispatcher.sh` для `antigravity` — просто сообщение "откройте чат".
   - `agy_runner.sh` примитивный (старый git checkout, нет skills, нет worktree, нет CI, нет памяти).

3. **Grok (самый сильный раннер) намеренно отфильтровывается**
   - `grok_worker.sh` жёстко пропускает всё, где `preferred_agent` не `auto`/`grok`.
   - Только один systemd-сервис (`agentforge-worker.service`) реально работает.

4. **Jules имеет воркер, но почти не получает задач**
   - `jules_worker.sh` существует и относительно нормальный, но:
     - Не установлен как systemd сервис.
     - Маршрутизация на него крайне узкая (`simple + test/docs`).

5. **Нет единой концепции "профиля исполнения"**
   - Antigravity задуман как "архитектор + полный доступ", но не имеет соответствующей инфраструктуры запуска.
   - Смешение ролей: иногда на Antigravity летят тяжёлые исполнительские задачи (scrape, web, mcp).

### Фактическое распределение (по логам и коду)

- Большинство сложных задач создаются с `preferred_agent: "antigravity"`
- После диспетчеризации они часто повисают или требуют ручного вмешательства
- Существует костыль `reassign.py`, который люди запускают, чтобы вернуть задачи на Grok

---

## 2. Цели рефакторинга

1. **Grok** должен быть основным автоматическим исполнителем для большинства задач (особенно Rust, рефакторинг, интеграции, краулеры).
2. **Jules** должен реально использоваться для задач, где нужен GitHub PR + тесты + документация.
3. **Antigravity** должен использоваться осознанно:
   - Либо как "человек + мощные инструменты" (ручной режим в IDE).
   - Либо получать полноценный мощный runner (если хотим частичную автоматизацию).
4. Уменьшить количество "зависших" задач из-за неправильной маршрутизации.
5. Сделать routing более явным, предсказуемым и легко настраиваемым.
6. Сохранить возможность "эскалации" (grok → antigravity при необходимости глубокого архитектурного решения).

---

## 3. Предлагаемая новая модель

### Три чётких режима работы агента

| Агент        | Режим исполнения          | Когда использовать                          | Требования к инфраструктуре |
|--------------|---------------------------|---------------------------------------------|-----------------------------|
| **grok**     | Полная автоматизация      | 80-90% задач (код, рефакторинг, интеграции) | Лучший runner + skills + CI |
| **jules**    | Автоматизация + PR        | Задачи с акцентом на тесты, docs, PR        | Отдельный worker + GitHub   |
| **antigravity** | Глубокий анализ / Архитектура / Human-in-the-loop | Только когда явно нужно архитектурное решение или review | Либо ручной чат, либо отдельный мощный runner (опционально) |

### Принципы маршрутизации (новые)

1. **По умолчанию** — `grok` (самый зрелый и быстрый исполнитель).
2. Явный `preferred_agent` уважается, но с предупреждением в логах, если это antigravity.
3. Jules выбирается только по конкретным признакам (наличие тегов `pr`, `github`, `tests-heavy`, `documentation` + medium/complex).
4. Antigravity — только при явном указании **или** специальном теге `deep-analysis` / `architecture-review`.
5. Сложность задачи (`complexity`) больше не является главным триггером для Antigravity.

### Фаза 1 — Исправление маршрутизации (самое важное)

**Файлы:**
- `task_queue.py` — полностью переписать `resolve_agent()`
- `dispatcher.sh` — улучшить обработку antigravity + добавить логирование
- `agent_cards.json` — обновить `routing_rules`

**Конкретные изменения:**

1. Новая логика `resolve_agent()`:
   ```python
   # Приоритет:
   # 1. Явный preferred_agent (кроме antigravity — с флагом "needs_review")
   # 2. Теги, явно указывающие на Jules (pr, github-pr, documentation-heavy)
   # 3. Тег "deep-analysis" / "architecture-decision" / "critical-review" → antigravity
   # 4. Всё остальное → grok
   ```

2. В `dispatcher.sh` для `antigravity`:
   - Не просто echo, а:
     - Пометка задачи специальным статусом/тегом `requires_human_antigravity`
     - Отправка уведомления (webhook / dashboard)
     - Опционально: создание "shadow" задачи на grok с пометкой "предварительный анализ"

3. Уменьшить количество скриптов `create_*.py`, которые хардкодят `antigravity`.

### Фаза 2 — Улучшение инфраструктуры запуска

**Вариант A (рекомендуемый на старте):**
- Сделать Antigravity преимущественно **ручным/полуручным** режимом.
- Улучшить `agy_runner.sh` → переименовать в `antigravity_runner.sh` с базовой поддержкой worktree + skills (но без полной силы grok_runner).

**Вариант B (если хотим частичную автоматизацию Antigravity):**
- Создать полноценный `antigravity_runner.sh` на базе `grok_runner.sh` с дополнительной инъекцией:
  - Содержимого `planlytasksko/AGENTS.md`
  - Таблиц защищённых параметров
  - Правил research subagent

**Jules:**
- Добавить systemd сервис `agentforge-jules-worker.service`
- Убедиться, что `jules_worker.sh` стабилен

### Фаза 3 — Улучшение observability и контроля

- Добавить в дашборд явное отображение "задачи, назначенные на Antigravity"
- Улучшить `reassign.py` → сделать более умным (с анализом, почему задача попала не туда)
- Добавить метрики в `/metrics` по распределению `assigned_agent`
- Логирование в dispatcher и workers: "Задача X была направлена на antigravity по причине Y — это может привести к ручной обработке"

### Фаза 4 — Долгосрочное (опционально)

- Ввести понятие `execution_profile` в задаче (вместо/в дополнение к `preferred_agent`).
- Сделать Antigravity способным **самому** создавать подзадачи на Grok/Jules (через API).

## 6. Риски и открытые вопросы

**Открытые вопросы (нужно решить перед началом Фазы 1):**

1. Хотим ли мы, чтобы Antigravity **в принципе** мог работать полностью автоматически, или он всегда должен быть "человек + инструменты"?
2. Готовы ли мы ослабить фильтр в `grok_worker.sh`, чтобы он мог брать задачи даже с `preferred_agent=antigravity` (с пометкой)?
3. Нужно ли сохранять возможность эскалации `grok → antigravity` внутри одной задачи?
4. Как именно Antigravity должен "возвращать" задачу в ферму после глубокого анализа (через создание подзадач на grok)?

**Риски:**
- Существующие зависшие задачи на antigravity (нужен migration скрипт).
- Люди привыкли создавать задачи сразу с `preferred_agent: "antigravity"`.
- Изменение поведения routing может неожиданно отправить важную задачу не туда.

## 7. Успешные критерии (как понять, что получилось)

- В дашборде ≥ 70% активных задач назначены на `grok` или `jules`.
- Количество зависших задач с `assigned_agent=antigravity` сильно снизилось.
- При создании сложной задачи по умолчанию она идёт на Grok (если не указан специальный тег).
- Jules реально берёт и закрывает задачи с PR.
- Нет необходимости регулярно запускать `reassign.py`.

## 9. Быстрая справка по новой маршрутизации (Quick Reference)

**После рефакторинга (Фаза 1, июнь 2026):**

| Ситуация                              | Какой агент получит задачу | Почему |
|---------------------------------------|----------------------------|--------|
| Новая задача без `preferred_agent`    | **grok**                   | Grok — основной исполнитель |
| Тег `deep-analysis`, `architecture-decision`, `critical-review` | **antigravity** | Только для настоящего глубокого архитектурного разбора |
| Теги `pr`, `documentation`, `tests-heavy` | **jules**             | Jules специализируется на PR и доках |
| Явно указано `preferred_agent: "antigravity"` | **antigravity**     | Уважаем явное желание человека |
| Задача долго висит на antigravity     | Используй `fix_antigravity_tasks.py` | Миграционный инструмент |

**Полезные команды после рефакторинга:**

```bash
# Диагностика текущего состояния
python agentforge/show_agent_stats.py

# Миграция зависших antigravity задач
python agentforge/fix_antigravity_tasks.py --dry-run
python agentforge/fix_antigravity_tasks.py

# Переустановка всех сервисов
bash agentforge/install_services.sh
```

**Полезные файлы:**
- `AGENTFORGE_ROUTING_AND_EXECUTION_REFACTOR_PLAN.md` — полный план и история изменений
- `show_agent_stats.py` — текущая статистика по агентам
- `fix_antigravity_tasks.py` — инструмент исправления старых задач
- `reassign.py` — более простой вариант миграции

После Фазы 1 рекомендуется прогнать существующие pending задачи через `reassign.py` (или новую версию).
